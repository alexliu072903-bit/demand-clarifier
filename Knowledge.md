##  Full Output Enforcement
---

## 核心原则

把每一个任务都当作生产级任务处理。**不完整的输出等于损坏的输出。**

不要为了简洁而牺牲完整性。用户要求完整文件，就交付完整文件。用户要求5个组件，就交付5个组件。没有例外。

---

## 禁止的输出模式

以下模式是硬性失败，绝对不能出现：

**代码块中禁止：**

- `// ...`
- `// rest of code`
- `// implement here`
- `// TODO`
- `/* ... */`
- `// similar to above`
- `// continue pattern`
- `// add more as needed`
- 用裸 `...` 代替省略的代码

**文字描述中禁止：**

- "如果需要我可以继续"
- "需要更多细节请告诉我"
- "为了简洁起见"
- "其余部分遵循相同的模式"
- "剩余部分类似"
- "以此类推"（用来替代实际内容时）
- "这部分留给你自己实现"

**结构性偷懒禁止：**

- 请求完整实现时只输出骨架
- 只展示第一段和最后一段，跳过中间内容
- 用一个示例加描述来替代重复逻辑
- 描述代码应该做什么，而不是真正写出来

---

## 执行流程

1. **定scope** — 读完完整请求，数清楚有多少个独立交付物（文件、函数、章节、回答）。锁定这个数字
2. **完整构建** — 每一个交付物都要完整生成，不留草稿，不说"你可以之后扩展"
3. **交叉核验** — 输出之前，重读原始请求，对比交付物数量和 scope 数量，有缺漏就补上再输出

---

## 处理长输出

当回复接近 token 上限时：

- 不压缩剩余章节来强行塞进去
- 不跳到结论
- 以完整的断点结尾（一个函数结束、一个文件结束、一个章节结束）
- 在断点处输出：

```
[已暂停 — 已完成 X / Y。发送「继续」从以下位置恢复：[下一节名称]]
```

收到"继续"后，从中断处精确继续。不重复摘要，不重复已有内容。

---

## 输出前快速核验

输出任何内容之前，确认以下全部通过：

- [ ] 上面禁止列表中的模式没有出现在任何地方
- [ ] 用户请求的每一项都存在且完整
- [ ] 代码块包含真正可运行的代码，而不是对代码的描述
- [ ] 没有任何内容因为节省空间而被缩短
## 一、平台架构五层模型（完整版）

用于在需求澄清阶段识别被忽略的系统影响面。收到需求时，在脑子里逐层过一遍。

### 第一层：数据层

**关注点**：实体、关系、存储模型、多租户策略、数据隔离、访问控制

关键问题：

- 这个需求会新增哪些数据实体？
- 多租户场景下，数据如何隔离？（共享 Schema / 独立 Schema / 独立数据库）
- 谁有权访问这些数据？权限边界在哪里？

> 数据层的决策最难逆转，在需求阶段就要把潜在冲突暴露出来。

### 第二层：业务逻辑层

**关注点**：规则、权限、工作流、状态机

关键问题：

- 这个功能涉及哪些业务规则？规则有没有例外情况？
- 状态变化是什么？什么触发状态转移？
- 不同角色的权限边界在哪里？

### 第三层：API 集成层

**关注点**：接口设计、Webhook、版本策略、生态扩展性

关键问题：

- 这个能力需要对外暴露 API 吗？
- 是否涉及第三方系统的集成？
- API 变更会影响哪些已有的调用方？

### 第四层：应用层

**关注点**：各端交互模型、性能要求、用户体验

关键问题：

- 这个功能在哪个端使用？（Web / Mobile / Admin）
- 性能要求是什么？用户的等待容忍度是多少？
- 有没有复杂的多步骤交互需要特别设计？

### 第五层：可观测层

**关注点**：日志、指标、事件追踪、告警

关键问题：

- 上线后怎么知道这个功能是否正常工作？
- 需要追踪哪些用户行为事件？
- 出问题时如何快速定位？

## 二、零到一阶段完整框架

### 核心原则

**先服务一侧。** 多边平台几乎都需要先选定一侧用户服务好，再通过他们的价值吸引另一侧。哪一侧被服务好之后，会主动拉动另一侧加入？

**先做应用，再做平台。** 第一版应该是一个解决具体问题的聚焦应用，平台架构等核心价值假设验证后再叠加。

**手动优先于自动化。** 在构建自动化之前，先手动服务前10-50个用户。从中学到的真实工作流，比任何前期设计都更有价值。

**激进压缩范围。** 用户体验到核心价值，最少需要哪些功能？其余的都是干扰。

### 零到一启动检查清单

- [ ] 核心用户问题已用真实用户验证（不是假设）
- [ ] 主要角色的端到端流程已完整跑通
- [ ] 多租户基础架构已到位（即使当前只有一个租户）
- [ ] 基础可观测性已建立：能看到用户在做什么
- [ ] 北极星指标已定义且可测量
- [ ] "毕业标准"已明确：这个阶段的成功是什么样的？

## 三、一到十阶段完整框架

### 核心杠杆点

**深化数据模型。** 早期平台数据模型往往很薄，只为验证概念。这个阶段投资更丰富的实体和关系，解锁新能力而不需要新界面。

**投资 API 层。** 平台价值通过集成和扩展复利增长。现在建好 API 层，未来几年都会持续受益。

**系统化手动流程。** 把早期为客户手动完成的事情（新手引导、配置模板、权限设置）变成系统自动完成。

**服务超级用户。** 最活跃的用户正在触碰 v1 的边界。为他们构建批量操作、高级筛选、可配置工作流，通常对所有用户都有价值。

**强化可观测层。** 10倍用户量下，1倍时看不到的问题会变成危机。在问题爆发之前建好结构化日志和告警体系。

## 四、北极星指标框架（完整版）

### 什么是好的北极星指标

- 移动时代表用户在获得真实价值（不只是登录）
- 全团队都能理解
- 可以被拆解为下级指标，每个团队都能影响

### 分解树结构

```
北极星指标：[e.g. 周活跃团队数]
  └── 获客：本周新激活团队数
  └── 激活：完成新手引导并创建第一个 X 的团队
  └── 留存：上周活跃且本周也活跃的团队
  └── 扩展：本周新增成员或功能使用的团队
```

### 常见陷阱

- **用活跃度代替价值**："DAU"不等于用户在获得价值，要找到能证明价值交换发生的指标
- **指标太多**：超过一个北极星就等于没有北极星
- **无法分解**：如果团队不知道自己的工作如何影响这个指标，它就没有用

## 五、多租户与角色设计检查清单

适用于任何涉及多用户、多组织的产品需求澄清。

### 租户隔离

- [ ] 不同组织的数据是否完全隔离？
- [ ] 隔离策略是什么？（共享表加 tenant_id / 独立 Schema / 独立数据库）
- [ ] 是否有跨租户的场景？（如基准对比、共享模板）如果有，数据边界怎么设计？

### 角色权限

- [ ] 产品中有哪些角色？（超级管理员 / 租户管理员 / 普通用户 / 只读用户）
- [ ] 每个角色能看到什么？能操作什么？
- [ ] 权限逻辑放在哪里？（前端隐藏 / API 校验 / 数据库行级别）
- [ ] 权限变更时，已发出的邀请链接/已创建的会话如何处理？

## 六、需求澄清访谈问题库

### 通用问题（任何需求都适用）

**关于用户**

- 谁是这个功能的主要使用者？他们的角色是什么？
- 他们现在用什么方式解决这个问题？
- 他们在哪个场景下会触发这个需求？

**关于范围**

- 这个版本明确不包括什么？
- 有没有"看起来相关但这次不做"的功能？
- 如果只能做一件事，哪件事最关键？

**关于成功标准**

- 上线后怎么判断这个功能成功了？
- 两周后回来看，什么数据变化说明方向对了？
- 什么情况下我们会说"这次失败了，需要调整"？

**关于约束**

- 技术栈有什么限制？
- 时间窗口是多久？
- 有没有必须兼容的已有系统或数据结构？

### 探索型需求专用问题

- 你怎么知道这是真实需求，而不是你觉得他们有这个需求？
- 最小可验证的版本是什么？需要多久能跑通？
- 如果这个做成了，3年后是什么样子？

### 执行型需求专用问题

- 这个需求背后的业务目标是什么？
- 如果不做这个，会发生什么？
- 有没有更简单的方式达到同样的目标？

## 七、设计文档摘要模板

每次完成需求澄清后输出，作为进入 Blueprint 阶段的输入文档。

```
【设计文档摘要】

问题定义：
[一句话说清楚：谁遇到了什么问题，现在的解决方式有什么不足]

目标用户：
[具体的用户角色和使用场景]

核心价值：
[用户用了这个产品之后，他们的什么事情变得更好了？]

推荐方案：
[选择的实现路径，以及为什么不选其他路径]

明确不包括：
[这个阶段不做的事情，以及原因]

成功指标：
[上线后用什么数据来判断成功]

第一步行动：
[今天可以开始做的最小一件事]

待验证假设：
[进入开发前还需要验证的关键假设]
```
